description: High-signal PR review for pulumi-garage (Go provider)
---

Provide a code review for the given pull request.

Review output must be terse and issue-focused. Do not praise the PR, narrate checks that passed, explain why code is correct, or offer "good change" commentary.

To do this, follow these steps precisely:

1. Check if the PR is closed, a draft, or not reviewable. If so, stop with a brief reason.

2. Read any CLAUDE.md files relevant to changed files.

3. Summarize the PR: title, description, changed files, main behavioral changes, risk areas.

4. Fetch existing review comments and threads on the PR. Identify:
   - Similar issues already flagged
   - Threads where a human already acknowledged the feedback
   - Stale comments on code that has changed since the earlier review
   - Bot-authored review threads that may now be fixed or obsolete

5. Launch review subagents in parallel. Each returns a list of candidate issues with:
   - Concise description
   - Affected file path and changed line
   - Why the issue is likely real

   Agent 1: CLAUDE.md compliance — audit changes for CLAUDE.md rule violations.
   Agent 2: Bug-focused — scan for obvious bugs in the diff. Flag only significant bugs.
   Agent 3: Behavior-focused — look for security issues, incorrect logic, regressions, and missing error handling in changed code.

   **CRITICAL: We only want HIGH SIGNAL issues.** Flag issues where:
   - The code will fail to compile (syntax errors, type errors, missing imports)
   - The code will definitely produce wrong results (clear logic errors)
   - The code introduces a clear security or regression bug
   - Clear, unambiguous CLAUDE.md violations

   Do NOT flag:
   - Code style or quality concerns
   - Potential issues that depend on specific inputs or state
   - Subjective suggestions

   If you are not certain an issue is real, do not flag it.

6. For each candidate issue, validate it against the actual code and context.

7. Deduplicate and prune:
   - Issues already covered by existing review comments
   - Issues acknowledged by a human
   - Issues fixed in the latest diff
   - Duplicate findings
   - Findings not on changed lines

   Also identify bot-authored threads eligible for resolution (fixed, outdated, or acknowledged).

8. Classify remaining issues:
   - **Blocking**: correctness, security, regression, data loss
   - **Non-blocking**: actionable but not merge-blocking

   Drop anything that is praise, a follow-up idea, a readability suggestion with no concrete risk, or an observation that does not require author action.

9. If no actionable issues remain, approve with one short sentence.

10. If actionable issues remain, post one inline comment per issue:
    - Brief description, why it matters, exact line reference
    - Cite the relevant CLAUDE.md rule when applicable
    - Keep comments concise and actionable

11. Resolve eligible stale bot-authored threads before submitting the final review.

12. Submit exactly one final review:
    - `REQUEST_CHANGES` when at least one blocking issue exists
    - `APPROVE` otherwise
    - Do not use `COMMENT` as the final review state

False positives to ignore:

- Pre-existing issues
- Something that appears buggy but is actually correct
- Pedantic nitpicks
- Issues a linter will catch
- Differences only in files classified as generated by `.gitattributes`
- Explanations that a change is acceptable as-is
- Non-blocking observations that do not require author action
